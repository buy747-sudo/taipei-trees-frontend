# taipei-trees.org 平台願景與功能架構 — 多視角規劃

> 建立日期：2026-06-11
> 目標：台北市最完整的樹木平台，世界一級標準
> 性質：執行架構文件 — 每項功能標註優先級與技術可行性
> 前提約束：純靜態前端（無框架）、CDN 載入、tree-app API 後端、公園樹待授權、`gov_risk_level` 不對外

---

## 一、設計總綱

世界一級的城市樹木平台，看三件事：

1. **資料完整且活的** — 不只是一份匯入的清冊，而是持續被普查、評估、通報更新的活資料庫。我們已經做到這一點，這是與全球多數「樹木地圖」最大的差異（紐約 NYC Street Tree Map、倫敦 London Tree Map 都只是靜態快照）。
2. **每個角色打開網站 30 秒內看到自己要的東西** — 民眾看樹、巡檢員看工作、處長看風險、市長看政績。
3. **圖表會說故事** — 不是把數字丟出來，而是回答「所以呢？」：碳匯換算成幾座大安森林公園、風險樹處理進度像進度條一樣一眼看完。

對標案例（功能參考，不照抄）：
- **NYC Street Tree Map**：民眾認養、生態效益貨幣化（每棵樹年省 $XX 美元）
- **Singapore trees.sg**：開花通知、樹木故事、300萬棵樹規模的流暢地圖
- **Melbourne Urban Forest Visual**：給每棵樹 email 的市民互動、樹齡壽命預測圖
- **i-Tree（USDA）**：生態效益計算的學術標準（我們 benefits.js 已採用同源方法）

## 三、圖表技術選型

目前全站零圖表 library，統計用表格呈現。要做世界一級的視覺化，建議：

**選 Apache ECharts（CDN 版）**，理由：
- 符合「無 build step、CDN 載入」的架構鐵律（與 Leaflet、jsQR 同模式）
- 繁中文件完整、台灣開發者熟
- 熱力圖、地圖著色、趨勢線、儀表板全部內建
- 手機觸控支援好（符合手機優先原則）
- Apache 2.0 授權，政府案無疑慮

地圖上的視覺化（路段著色、熱力圖）直接用 Leaflet 原生能力＋ GeoJSON，不需要額外 library。

**設計原則**：
1. 每張圖表都要回答一個問題，標題就是那個問題（「哪一區的樹最多？」不是「行政區統計」）
2. 數字要翻譯：碳匯講「相當於 N 趟台北-高雄來回飛機」，不只講公噸
3. 戶外高對比配色沿用現有原則
4. 圖表都附「資料截至 YYYY-MM-DD」與來源標註（公信力）

## 五、跨領域架構事項

### 權限分層（現有 5 角色之上的最終藍圖）

```
公開            民眾：地圖、詳情、生態效益、綠資產儀表板、通報
─────────────────────────────────────────────
surveyor        普查 wizard、自己的紀錄、今日清單
inspector       + 評估、風險圖層、處理回報
contractor_admin + 公司儀表板、派工、報表
gov_viewer      + 全市總覽、廠商比較（唯讀）
gov_editor      + 通報轉派、狀態審核
platform_admin  全部 + 帳號管理 + 數據總覽
```

### 資料閉環（平台的核心競爭力）

```
普查 ──> 樹籍資料 ──> 民眾查詢
            │              │
評估 ──> 風險標記 ──> 處理回報 ──> 圖示更新（閉環）
            │
民眾通報 ──> 後台收件 ──> 轉派廠商 ──> 處理 ──> 結案（閉環）
```

每個箭頭都要留 audit log；狀態變更不互相覆蓋。

### 不做的事（重申鐵律）

- ❌ 不引入框架、不做 SSR、不自建後端
- ❌ 不公開 `gov_risk_level`、`gov_health_score`
- ❌ 公園樹授權未明前不上線
- ❌ 民眾登入功能（收藏用 localStorage 即可）
- ❌ 通報不做成 1999 替代品、不承諾處理時程

## 六、開放問題（待 Nash 決定）

1. **P0 兩項是否要在 6/20 前做？** 風險低但仍是新 code，另一個選擇是評選只用現有功能＋投影片講願景。
2. **普查進度地圖的「路段」單位**：用行政區著色（資料現成、粗）還是路段著色（需要路段邊界 GeoJSON，可能要從市府圖資取得）？建議先做行政區版。
3. **政府帳號**：評選 demo 要不要現場開一個 gov_viewer 帳號給委員操作？需要的話 P1 的全市總覽要提前。
4. **綠資產儀表板的訪客數**：要不要公開「本站查詢次數」？對市長視角是亮點，但數字小的時候反而扣分，建議累積到有規模再放。
